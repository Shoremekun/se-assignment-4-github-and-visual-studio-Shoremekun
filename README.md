GitHub Actions:

Explain what GitHub Actions are and how they can be used to automate workflows. Provide an example of a simple CI/CD pipeline using GitHub Actions. Answer: Github actions are features that allows you to automate workflows directly within your GitHub repository. They enable you to define workflows that automatically execute tasks in response to specific events, such as code pushes, pull requests, or schedule triggers. GitHub Actions are used for continuous integration (CI), continuous deployment (CD), and other automation tasks. Github actions:  1. Workflows: A workflow is an automated process defined in a YAML file located in the `.github/workflows` directory of your repository. It consists of one or more jobs. 2. Jobs: Jobs are units of work within a workflow. Each job runs on a specified runner and can include multiple steps. 3. Steps: Steps are individual tasks within a job. They can run commands, use actions, or perform other operations. 4. Actions: Actions are reusable pieces of code that perform specific tasks. They can be created by GitHub or third parties and are used within workflows to perform tasks such as checking out code, running tests, or deploying applications. 5. Runners: Runners are servers that execute the jobs defined in your workflows. GitHub provides hosted runners, but you can also set up self-hosted runners. Examples: 1. Create a workflow file: Create a YAML file in your repository under `.github/workflows/ci.yml`. name: CI Pipeline
   on:
     push:
       branches:
         - main
     pull_request:
       branches:
         - main

   jobs:
     build:
       runs-on: ubuntu-latest

       steps:
       - name: Checkout code
         uses: actions/checkout@v3

       - name: Set up Node.js
         uses: actions/setup-node@v3
         with:
           node-version: '14'

       - name: Install dependencies
         run: npm install

       - name: Run tests
         run: npm test
2. Explanation of the Workflow: Defines the name of the workflow. Specifies the events that trigger the workflow. In this case, the workflow runs on pushes and pull requests to the `main` branch. Defines the jobs to be executed in the workflow. This example has a single job named `build`. Specifies the runner environment. `ubuntu-latest` is used here. Lists the steps in the job: Uses the `actions/checkout` action to check out the repository’s code. Uses the `actions/setup-node` action to set up Node.js. Runs `npm install` to install project dependencies. Runs `npm test` to execute tests. 3.How It Works: When you push code to the `main` branch or create a pull request targeting `main`, GitHub Actions triggers this workflow. The workflow checks out your code, sets up Node.js, installs dependencies, and runs tests. If any step fails, the workflow will indicate an error, helping you catch issues before merging code or deploying.
